## 2. 完整阅读后的结论

### 2.1 目前文档的优点

现有文档已经具备比较完整的基础骨架：

- 有清晰的产品目标
- 有功能拆分和任务拆分
- 有技术架构方向
- 有基础隐私与合规意识

这些内容足够支撑进入正式设计和开发阶段。

### 2.2 目前文档的主要问题

但如果严格按当前文档直接开发，会在 4 个点上产生返工：

#### 问题 P1：`颜色标记` 的定义不够准确

当前文档把“颜色标记”描述为“显示在标签页上”，但 Chrome 扩展无法稳定直接改浏览器原生 tab 背景色。

这意味着：

- 不能把 T-202 理解为“给 Chrome 原生标签上色”
- 必须把颜色能力改为“视觉颜色表达”

建议实现方式：

- favicon 色徽标
- 标题前缀
- 扩展内色签
- 页面内角标

#### 问题 P2：`chrome.tabs API 存储标签标记` 的表述不准确

文档中多次写到“通过 chrome.tabs API 存储标记”，这不适合作为正式存储方案。

更合理的方案是：

- 用 `chrome.storage.local` 保存扩展侧状态
- 用 `tabId` 作为运行期主键
- 通过内容脚本把状态投射到页面表现层

也就是说：

- `chrome.tabs` 用于查询和切换标签
- `chrome.storage.local` 才是正式持久化层

#### 问题 P3：快捷键“插件内可配置”与 Chrome 实际能力不一致

现有文档假设可以在 options 页面直接做快捷键编辑、验证、保存。

实际规划应改为：

- 在 `manifest.json` 中声明默认快捷键
- 在 options 页面展示命令说明
- 提供跳转到 `chrome://extensions/shortcuts` 的入口

否则会在 T-603 和 T-504 上产生实现偏差。

#### 问题 P4：内容脚本权限模型尚未落地

文档当前只写了：

- `tabs`
- `storage`
- `activeTab`
- `commands`

但如果首版希望做到以下任一能力：

- 页面刷新后自动恢复标记
- 修改标题前缀
- 覆盖 favicon
- 页面内显示角标

则至少还需要考虑：

- `scripting`
- `host_permissions`

否则只能做“当前主动触发一次”的弱体验版本。

## 3. 产品重新收敛

### 3.1 首版真正要解决的问题

我建议把首版产品问题定义为：

> 当用户同时打开很多网页时，TabMarker 帮助用户用统一视觉标记快速识别和跳转当前打开的页面。

这个定义有两个重点：

- 只解决“打开中的页面管理”
- 先解决“快速识别 + 快速切换”，不扩展到历史管理、云同步、自动规则

### 3.2 首版产品定位

TabMarker 不是一个“浏览器主题插件”，而是一个：

> 本地优先的标签视觉标记与快速定位工具

### 3.3 MVP 目标

MVP 要保证 3 件事：

1. 用户能在 10 秒内给当前页完成标记
2. 用户能在弹窗里快速找到并跳转已标记标签
3. 用户刷新页面后，标记大概率还能恢复

## 4. 推荐 MVP 范围

### 4.1 必做能力

#### M1 当前标签打标
- 颜色
- emoji
- 文本
- 组合标记

#### M2 当前标签清除
- 清除状态
- 恢复标题
- 移除页面角标
- 移除 favicon 覆盖

#### M3 预设组合
- 保存预设
- 删除预设
- 一键应用预设
- 最多 5 个预设

#### M4 已标记标签列表
- 展示当前窗口所有已标记标签
- 点击切换
- 当前标签高亮

#### M5 自动恢复
- 当前标签刷新后自动恢复
- Service Worker 重启后仍能读回状态

### 4.2 建议做能力

#### M6 页面内角标
- 用于进入页面后再次确认标记状态
- 可以默认开启，设置中允许关闭

#### M7 受限页面降级提示
- Chrome 内置页、商店页等不可注入页面需要明确提示

### 4.3 不建议首版做

- 插件内快捷键编辑器
- 复杂 emoji 搜索
- 跨窗口批量操作
- 云同步
- 自动规则打标

## 5. 视觉与交互规划

### 5.1 设计原则

- 一屏完成核心任务
- 优先即时反馈
- 风格可以新，但动效要轻
- 低频操作放 options，高频操作留 popup

### 5.2 Popup 结构

建议采用 5 个区域：

#### A. 头部状态
- 当前站点 favicon
- 页面标题简写
- 当前状态标签

#### B. 实时预览
- 模拟标签标题
- 色签样式
- 应用反馈

#### C. 标记编辑器
- 颜色网格
- emoji 选择器
- 文字输入
- 清除 / 应用按钮

#### D. 预设区
- 已保存预设卡片
- 保存当前为预设

#### E. 已标记列表
- 当前窗口已标记标签
- 快速跳转

### 5.3 Options 结构

建议仅保留低频能力：

- 预设管理
- 页面角标开关
- 快捷键说明
- 权限说明
- 受限页面说明

### 5.4 视觉方向

建议沿用深色、玻璃感、轻霓虹路线：

- 深色背景
- 当前选中颜色驱动主高亮
- 卡片式结构
- 120ms~180ms 轻过渡

## 6. 技术落地规划

### 6.1 推荐技术基线

- Manifest V3
- 原生 HTML / CSS / JavaScript
- 模块化目录组织
- `chrome.storage.local` 作为持久化层

### 6.2 推荐权限集

#### 首版推荐

```json
{
  "permissions": [
    "storage",
    "tabs",
    "activeTab",
    "scripting",
    "commands",
    "contextMenus"
  ],
  "host_permissions": [
    "<all_urls>"
  ]
}
```

### 6.3 推荐模块结构

```text
src/
  background/
    main.js
    storage.js
    tab-state.js
    commands.js
    injection.js
  content/
    marker.js
  popup/
    popup.html
    popup.css
    popup.js
  options/
    options.html
    options.css
    options.js
  shared/
    constants.js
    validators.js
    utils.js
```

### 6.4 数据结构调整

#### 标记状态

```json
{
  "tabMarkers": {
    "123": {
      "tabId": 123,
      "windowId": 1,
      "url": "https://example.com",
      "title": "Example",
      "marker": {
        "color": "#22C55E",
        "emoji": "📘",
        "text": "DOC"
      },
      "original": {
        "title": "Example",
        "faviconUrl": "https://example.com/favicon.ico"
      },
      "updatedAt": 1775890000000
    }
  }
}
```

#### 预设状态

```json
{
  "savedPresets": [
    {
      "id": "preset_1",
      "name": "工作文档",
      "color": "#3B82F6",
      "emoji": "📘",
      "text": "DOC"
    }
  ]
}
```

### 6.5 表现层方案

首版标记建议使用四层表达：

#### L1 扩展内状态
- 永远可见
- 是唯一真实数据源

#### L2 标题前缀
- `emoji + [text] + 原始标题`

#### L3 favicon 色徽标
- 用颜色补足视觉区分

#### L4 页面内角标
- 进入页面后能确认当前标签被标记

### 6.6 降级策略

对于以下页面：

- `chrome://*`
- `edge://*`
- Chrome Web Store
- 部分特殊安全页面

只提供：

- 扩展内列表标记
- 状态提示

不承诺标题、favicon、页面角标生效。

## 7. 重新编排后的开发阶段

### 阶段 0：开发前对齐

目标：先把最容易返工的点拍板。

- 是否接受“颜色 = 色徽标，不是原生 tab 背景色”
- 是否接受申请 `<all_urls>`
- 是否保留页面内角标
- 首版是否包含已标记列表

### 阶段 1：技术骨架

目标：让扩展能跑起来。

任务：

- 建立目录结构
- 搭建 manifest
- 初始化 popup / background / options / content
- 配置共享常量和消息协议

交付物：

- 可加载的扩展骨架
- popup 空壳页
- background 正常运行

### 阶段 2：状态与注入最小闭环

目标：完成“应用标记 -> 页面生效 -> 存储成功”的基础闭环。

任务：

- 实现 `storage.js`
- 实现 `tab-state.js`
- 实现当前标签获取
- 实现标题前缀注入
- 实现清除逻辑

交付物：

- 当前标签可以打标和清除
- 刷新后可恢复标题前缀

### 阶段 3：可视化增强

目标：补齐颜色表达和页面内确认能力。

任务：

- 实现 favicon 徽标
- 实现页面内角标
- 实现 popup 预览区

交付物：

- 颜色视觉表达完整
- 页面内可识别当前状态

### 阶段 4：预设与标签列表

目标：把产品从“可演示”推进到“可日用”。

任务：

- 保存 / 删除预设

交付物：

- 用户可快速复用常用标记
- 用户可直接从 popup 找回目标标签

### 阶段 5：快捷方式与设置页

目标：补齐低频操作入口。

任务：

- 增加 commands
- 增加 options 页面

交付物：

### 阶段 6：稳定性与发布准备

## 9. 风险清单

### 风险 R1：页面注入限制

影响：

- 某些页面无法显示角标或改写标题

应对：

- 明确降级策略
- 在 popup 中给出提示

### 风险 R2：favicon 恢复复杂度

影响：

- 页面原始 favicon 可能有多个 link 标签，恢复逻辑容易不完整

应对：

- 首版记录原始 favicon 信息
- 恢复失败时优先移除覆盖 favicon

### 风险 R3：Service Worker 生命周期

影响：

- 后台可能被回收，状态处理中断

应对：

- 所有核心状态写入 `chrome.storage.local`
- 每次 popup 打开都从存储层同步

### 风险 R4：权限敏感

影响：

- 若使用 `<all_urls>`，安装提示会更敏感

应对：

- 在产品说明和隐私文档中清楚解释用途
- 强调不上传数据、不采集浏览内容
