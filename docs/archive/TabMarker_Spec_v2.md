## 2. 对原始 PRD 的修正与补充

### 2.1 需要先对齐的关键现实约束

原始 PRD 的方向正确，但有两处需要在开发前修正：

1. Chrome 扩展无法稳定直接修改浏览器原生标签页背景色。
- 结论：`颜色标记` 不能定义为“修改 Chrome 原生 tab strip 的颜色”。
- 建议改为：通过 `标题前缀`、`favicon 徽标`、`扩展内列表色签`、`当前页页面角标` 共同组成“颜色标记”的视觉表达。

2. Chrome 扩展不能在扩展内完全自定义快捷键管理。
- 结论：`快捷键设置界面` 不能定义为“插件内直接改写系统快捷键”。
- 建议改为：插件展示当前快捷键、提供说明、并跳转到 `chrome://extensions/shortcuts` 进行修改。

### 2.2 产品真实能力边界

TabMarker 的本质不是“改造 Chrome 原生标签 UI”，而是：

- 为当前页面注入可恢复的标签标记
- 为打开中的标签页建立本地可视化管理层
- 通过扩展弹窗和快捷操作降低多标签检索成本

换句话说，产品应定义为：

> 一个本地优先的 Chrome 标签视觉标记与快速定位工具。

## 3. 产品定义

### 3.1 产品定位

TabMarker 是一款运行在 Chrome / Chromium 浏览器中的本地扩展，用于帮助用户对当前打开页面进行轻量标记，并通过统一的视觉体系快速识别、切换和清理标签。

### 3.2 产品目标

- 让用户在多标签场景下 3 秒内找到目标标签
- 让用户在 10 秒内完成一次标签标记
- 保证核心操作无需离开当前页面即可完成
- 所有数据本地存储，不依赖外部服务

### 3.3 核心价值

- 快速识别：给标签增加稳定的视觉特征
- 快速操作：一键复用常用标记
- 快速恢复：刷新后自动恢复可恢复的标记
- 低打扰：不改变用户原有浏览习惯

### 3.4 目标用户

#### 用户类型 A：重度知识工作者
- 常开 15~50 个标签页
- 经常在多个文档、后台系统、搜索结果之间切换
- 需求重点：快速区分“正在处理 / 待阅读 / 待回访”

#### 用户类型 B：开发者
- 同时打开 API 文档、控制台、管理后台、代码托管平台
- 需求重点：快速定位环境页、文档页、调试页

#### 用户类型 C：运营 / 研究人员
- 同时处理表格、报表、素材、信息收集页面
- 需求重点：为任务阶段打标签，降低切换成本

## 4. 使用场景与用户流程

### 4.1 核心场景

#### 场景 S1：当前标签快速标记
用户正在处理一个重要页面，希望立刻给该标签添加醒目标记，方便稍后快速找到。

#### 场景 S2：一键应用常用组合
用户频繁使用同一组标记规则，例如：
- 红色 + `!` + `紧急`
- 蓝色 + `📘` + `文档`
- 绿色 + `DONE`

#### 场景 S3：批量浏览中的快速定位
用户打开很多页面后，通过扩展弹窗查看已标记标签列表，并直接切换到目标页。

#### 场景 S4：刷新后自动恢复标记
页面刷新或重新加载后，用户希望标记仍然存在，避免重复设置。

### 4.2 标准用户流程

#### 流程 F1：首次使用
1. 用户安装扩展
2. 点击工具栏图标打开弹窗
3. 阅读 1 屏新手提示
4. 对当前标签设置颜色、emoji、文字
5. 点击应用
6. 当前页面立即出现标记效果

#### 流程 F2：常规使用
1. 用户打开某个标签页
2. 使用工具栏图标或快捷键打开弹窗
3. 修改标记
4. 点击应用
5. 关闭弹窗并继续工作

#### 流程 F3：应用预设
1. 用户打开弹窗
2. 在预设区点击某个常用组合
3. 系统直接应用并提示成功

#### 流程 F4：从已标记列表切换标签
1. 用户打开弹窗
2. 查看“已标记标签”列表
3. 点击某一项
4. 浏览器切换到对应标签

## 5. 范围定义

### 5.1 MVP 范围

MVP 只做“当前标签标记 + 预设复用 + 已标记列表 + 清除”。

#### 包含
- 当前标签设置颜色标记
- 当前标签设置 emoji 标记
- 当前标签设置文字 Tag
- 组合标记
- 清除当前标签标记
- 保存最多 5 个预设组合
- 一键应用预设
- 查看当前窗口中已标记标签列表
- 点击列表项跳转标签
- 标记状态本地持久化
- 页面刷新后的标记恢复

#### 不包含
- 跨设备同步
- 云端账号
- AI 自动打标
- 自动规则打标
- 批量多选标签操作
- 插件内可编程快捷键编辑器

### 5.2 P1 范围

- 右键菜单快速标记
- 选项页管理预设
- 支持最近使用 emoji
- 支持标签过滤和搜索
- 支持“只看已标记标签”

### 5.3 P2 范围

- 标签分组联动
- 域名规则自动应用预设
- 导入导出本地配置

## 6. 功能需求明细

### 6.1 当前标签标记

#### FR-001 标记编辑器
- 用户可为当前活动标签设置颜色、emoji、文字
- 三种元素可单独使用，也可组合使用
- 至少选择或输入一种内容才允许应用

#### FR-002 颜色标记
- 提供 12 个预设颜色
- 支持 1 个自定义颜色输入
- 颜色用于以下区域：
  - 扩展弹窗中的预览条
  - 已标记标签列表的色签
  - 页面内角标或视觉标识
  - favicon 徽标背景色

#### FR-003 Emoji 标记
- 提供常用 emoji 面板
- MVP 不做复杂分类搜索，可先提供 24 个高频 emoji
- emoji 用于标题前缀和列表展示

#### FR-004 文字 Tag
- 长度限制 12 个可见字符
- 超过限制时禁止提交并给出提示
- 文本内容进行转义，禁止 HTML 注入

#### FR-005 组合预览
- 用户在编辑器内修改任一元素时，预览区域实时更新
- 预览应包含：
  - 标题效果预览
  - 色签效果预览
  - 当前选择值

#### FR-006 应用标记
- 点击应用后，系统立即将标记写入本地状态
- 若当前页面支持内容注入，则应用以下展示：
  - 页面标题前缀
  - favicon 徽标
  - 页面角标
- 若当前页面不支持注入，则至少在扩展内列表中可见

#### FR-007 清除标记
- 用户可清除当前标签的标记
- 清除后应恢复原始标题和 favicon
- 若无法恢复 favicon，则降级为移除扩展生成的覆盖 favicon

### 6.2 预设组合

#### FR-101 保存预设
- 用户可将当前编辑内容保存为预设
- 保存时必须输入预设名称
- 最多保存 5 个预设
- 若达到上限，提示用户删除旧预设后再保存

#### FR-102 应用预设
- 用户点击预设卡片即可应用到当前标签
- 应用后弹窗内显示成功反馈

#### FR-103 管理预设
- MVP 允许删除预设
- 编辑预设可放到 P1

### 6.3 已标记标签列表

#### FR-201 列表展示
- 展示当前窗口下所有已标记标签
- 默认按最近激活时间倒序
- 每项显示：
  - 颜色色签
  - emoji
  - 文本 tag
  - 页面标题
  - 域名

#### FR-202 快速跳转
- 点击某项后切换到对应标签页
- 若标签已关闭，自动从列表中移除失效记录

#### FR-203 当前标签高亮
- 当前激活标签在列表中显示选中态

### 6.4 标记恢复

#### FR-301 刷新恢复
- 页面刷新后，若该标签有已保存标记，系统自动尝试恢复

#### FR-302 重启恢复
- 对普通网页标签，浏览器会话内可恢复
- 对已关闭标签不做历史恢复
- 对新开同 URL 页面默认不自动继承，避免误伤

### 6.5 快捷操作

#### FR-401 打开弹窗
- 支持点击扩展图标
- 支持默认快捷键打开

#### FR-402 快速应用预设
- 支持为前 3 个预设定义默认命令
- 若快捷键未设置，界面中展示“前往 Chrome 快捷键页配置”

## 7. 非功能需求

### 7.1 性能

- 弹窗首次打开时间小于 150ms
- 标记应用反馈时间小于 200ms
- 当前窗口 100 个标签时，已标记列表渲染时间小于 200ms

### 7.2 稳定性

- Service Worker 重启后，状态可重新读取
- 内容注入失败不应导致弹窗崩溃
- 对受限页面需提供明确提示

### 7.3 隐私

- 不发送网络请求
- 不收集行为分析数据
- 所有设置保存在本地

### 7.4 兼容性

- 支持 Chrome 114+
- 支持 Edge Chromium
- 不保证 `chrome://`、扩展页、Chrome Web Store 页面可注入

## 8. 交互设计方案

### 8.1 设计原则

- 一屏完成核心任务
- 所有关键动作 1 次点击可达
- 视觉风格新颖，但动画极轻
- 反馈优先于装饰

### 8.2 视觉方向

建议采用轻量玻璃拟态 + 霓虹高亮风格：

- 背景：深色渐变
- 重点色：由当前标签颜色决定
- 控件：圆角卡片、轻描边、弱阴影
- 动效：100~180ms 的过渡，不做复杂位移动画

### 8.3 Popup 信息架构

弹窗建议尺寸：`380 x 560`

#### 区块 A：顶部状态栏
- 产品名
- 当前页面 favicon
- 当前域名
- 当前标签是否已标记

#### 区块 B：实时预览区
- 模拟标签标题展示
- 色签条
- 应用结果提示

#### 区块 C：标记编辑区
- 颜色选择
- emoji 选择
- 文字输入
- 清除按钮
- 应用按钮

#### 区块 D：预设区
- 最近 / 常用预设卡片
- 保存当前为预设

#### 区块 E：已标记标签列表
- 当前窗口已标记标签
- 点击快速跳转

### 8.4 Options 页面信息架构

用于放置低频操作：

- 预设管理
- 快捷键说明与跳转入口
- 权限说明
- 支持页面限制说明

### 8.5 关键状态设计

#### 空状态
- 尚未设置任何标记
- 尚无任何预设
- 当前窗口暂无已标记标签

#### 成功状态
- 应用成功 toast
- 保存预设成功 toast

#### 异常状态
- 当前页面不支持注入
- 当前页面权限不足
- 预设数量超限
- 输入内容不合法

### 8.6 关键交互细节

#### 颜色选择
- 点击即选中
- 选中颜色立即驱动预览区变化

#### emoji 选择
- 点击后高亮
- 再次点击同一 emoji 可取消

#### 文字输入
- 实时统计剩余字符数
- 接近上限时颜色提醒

#### 应用按钮
- 无有效输入时置灰
- 点击后进入短暂 loading
- 成功后展示 1 秒内联反馈

#### 列表项
- hover 显示更强对比
- 点击整行切换标签

## 9. 技术方案

### 9.1 总体架构

采用 Chrome Extension Manifest V3 架构：

- `service_worker`
  - 负责标签状态管理
  - 负责命令监听
  - 负责标签更新与恢复调度

- `popup`
  - 负责当前标签编辑与预览
  - 负责预设应用
  - 负责展示已标记标签列表

- `options`
  - 负责低频设置与说明页

- `content script`
  - 负责修改页面标题
  - 负责注入 favicon 覆盖
  - 负责显示页面角标
  - 负责恢复原始页面信息

### 9.2 推荐实现策略

为了兼顾“效果可见”和“实现可靠”，建议采用四层标记策略：

#### L1 扩展内状态层
- 所有标签标记都保存在扩展状态中
- 这是唯一可信数据源

#### L2 标签标题前缀层
- 对可注入页面，将 `emoji + [text]` 写入 `document.title`
- 作用：用户在原生标签页上直接可见

#### L3 favicon 徽标层
- 基于原始 favicon 生成带背景色的小徽标
- 作用：补足“颜色标记”的感知

#### L4 页面角标层
- 在页面右上角注入可收起的小角标
- 作用：让用户进入页面后也能确认当前标签状态

### 9.3 为什么不用“直接修改标签颜色”

因为 Chrome 扩展没有稳定 API 去修改浏览器原生标签页背景色，所以颜色必须通过以下替代表达实现：

- favicon 徽标颜色
- 页面角标颜色
- 扩展内标签列表颜色
- 预览区颜色

### 9.4 权限方案

#### 方案 A：推荐方案

权限：
- `storage`
- `tabs`
- `activeTab`
- `scripting`
- `contextMenus`
- `host_permissions: ["<all_urls>"]`

优点：
- 标记恢复更稳定
- 可在页面刷新后自动重新注入
- 标题和 favicon 改写能力完整

缺点：
- 权限提示更敏感

#### 方案 B：弱权限方案

权限：
- `storage`
- `tabs`
- `activeTab`
- `scripting`

优点：
- 权限更克制

缺点：
- 只能在用户主动触发时处理当前页
- 无法稳定恢复所有页面标记
- 功能体验不一致

#### 推荐结论

如果产品目标是“真实可用”，建议选择方案 A。

### 9.6 模块拆分

#### `manifest.json`
- 权限、命令、入口页定义

#### `src/background/main.js`
- 初始化
- 监听命令
- 监听 `tabs.onUpdated`
- 监听 `tabs.onRemoved`
- 与 popup 通信

#### `src/background/storage.js`
- 读写 `chrome.storage.local`
- 数据校验
- 数据迁移

#### `src/background/tab-state.js`
- 维护当前会话中的 tab 标记状态
- 根据 tab 生命周期清理无效数据

#### `src/background/injection.js`
- 统一封装脚本注入与恢复逻辑

#### `src/content/content-marker.js`
- 执行标题改写
- 执行 favicon 覆盖
- 注入页面角标
- 提供恢复方法

#### `src/popup/popup.js`
- 获取当前 tab
- 读取状态
- 处理编辑器事件
- 渲染预设和标记列表

#### `src/options/options.js`
- 管理预设
- 展示快捷键配置入口

### 9.7 消息流

#### 应用标记
1. `popup` 收集用户输入
2. `popup -> background` 发送 `APPLY_MARKER`
3. `background` 持久化状态
4. `background` 调用注入器
5. `content script` 修改页面表现
6. `background -> popup` 返回结果

#### 清除标记
1. `popup -> background` 发送 `CLEAR_MARKER`
2. `background` 删除状态
3. `background` 通知 `content script` 恢复原始页面

#### 自动恢复
1. 浏览器触发 `tabs.onUpdated`
2. `background` 判断该 `tabId` 是否存在标记
3. 若存在则重新注入

### 9.8 受限页面降级策略

以下页面可能无法注入：

- `chrome://*`
- `edge://*`
- Chrome Web Store
- 部分扩展页
- 特殊 CSP 页面

降级策略：

- 标记状态仍记录在扩展内
- 在弹窗顶部显示“此页面仅支持扩展内显示”
- 已标记标签列表照常可见
- 不承诺原生标签可见效果

### 9.9 快捷键方案修正

Chrome 中快捷键的真实规则如下：

- 扩展可在 `manifest.json` 中声明默认快捷键
- 用户可在 `chrome://extensions/shortcuts` 修改
- 插件内不直接改写快捷键

因此设计方案应改为：

- 在 Options 页面展示“当前支持的命令”
- 提供“打开 Chrome 快捷键设置页”按钮
- 允许用户查看而非直接编辑快捷键

### 9.10 安全与数据策略

- 所有输入先进行长度和字符过滤
- 标题改写只写纯文本，不拼接 HTML
- favicon 覆盖只处理图片数据，不执行页面脚本
- 所有异常统一返回用户可理解的错误提示

## 10. 验收标准

### 10.1 MVP 验收

- 能对当前标签应用至少一种标记
- 组合标记应用后可立即看到反馈
- 页面刷新后可恢复标记
- 已保存预设可复用
- 已标记列表可跳转
- 清除后状态一致
- 遇到受限页面时有明确降级提示

### 10.2 技术验收

- 无未处理 Promise 异常
- Service Worker 重启后状态一致
- `chrome.storage.local` 数据结构稳定
- Popup 在空状态、异常状态、受限状态下不崩溃
